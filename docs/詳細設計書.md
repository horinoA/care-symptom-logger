# 詳細設計書 (フォルダ構成およびクラスの役割)

本プロジェクト（CareRecordDWH）は、DDD（ドメイン駆動設計）/オニオンアーキテクチャの思想に基づいて構築されています。依存関係は外側から内側（ドメイン）へ向かい、ドメイン層が他の層に依存しない設計となっています。

## フォルダ構成と役割

`tech.doshikawa.carerecord` 配下の主要なパッケージ構成は以下の通りです。

- **`presentation/`**: アプリケーションの入り口（外層）。LINEプラットフォームなどの外部からのリクエスト（Webhookなど）を受け取り、ビジネスロジックは持たずに `application/` 層に処理を委譲します。
- **`application/`**: ユースケースを実現する層。トランザクションの制御、ステートマシン（対話フロー）の進行、およびドメインオブジェクトを協調させてアプリケーションの目的を達成します。
- **`domain/`**: ビジネスロジックの核となる層（最内層）。データベースの構造や外部APIに依存せず、業務ルールそのものを表現する Entity、Enum（Type）、および Repositoryインターフェースを配置します。
- **`infrastructure/`**: 技術的な詳細を担当する層（外層）。Spring Data JDBCを用いたデータベースへのアクセス設定、外部API通信の設定など、泥臭い実装を担当します。
- **`common/`**: システム全体で共通して使用される設定クラス、ユーティリティ、例外処理クラスなどを配置します。

## 全クラスの役割一覧 
CareRecordDWH（介護記録データウェアハウス）の現在のクラス構成と各クラスの責務を整理したものです。オニオンアーキテクチャ（DDD）を採用し、依存関係が外側から内側（Domain層）に向かうように設計されています。

### 1. Presentation層
外部インターフェースからのリクエストを受け付ける層です。
* LineWebhookController 
  * LINEプラットフォームからのWebhookイベントを受信するControllerです。ビジネスロジックは持たず、リクエストのパースとApplication層（LineWebhookDispatcherService ）への委譲のみを行います。
### 2. Application層
ユースケースの進行と、外部とのメッセージングを管理する層です。
  #### サービス（Service）
  * LineWebhookDispatcherService 
    * 受信したWebhookイベントの種別（テキスト、Postbackなど）を判定し、処理を適切なHandlerへルーティングするディスパッチャです。
  * LineMessageService 
    * ユーザーへの返信メッセージ（テキストやFlex Message）の構築と、LINEMessaging APIを通じた送信処理を担います。
  * CareRecordApplicationService 
    * 介護記録の登録に関するユースケースを実現します。ステートマシンの最終フェーズにおいて、記録データの検証とデータベースへの保存（永続化）を指揮します。
  #### ハンドラー（PostbackActionHandler）
  * PostbackActionHandler  (Interface)
    * LINEのPostbackアクション（ボタン押下等）を処理するための共通インターフェースです。
  * 各種Handler実装クラス ( SelectWhoHandler ,  SelectSymptomCategoryHandler ,SelectDateHandler  等 計19クラス)
    * ユーザーのアクションごとに用意された処理クラスです。入力フェーズ（状態）の進行、セッションデータの更新、不正な遷移のブロック、および次のアクションを促すメッセージの返信処理を実行します。
    - `PostbackActionHandler.java`: Postbackアクションを処理するStrategyインターフェース。サポートするアクション判定と許可フェーズのガード機能（門番）を持ちます。
    - `SelectSymptomCategoryHandler.java`: 「症状カテゴリ選択」時のセッション更新と次フェーズへの進行を担当します。
    - `SelectSymptomDetailHandler.java`: 「具体的な症状選択」を処理し、継続確認へ進みます。
    - `BackToCategoryHandler.java`: 症状詳細からカテゴリ選択へ戻る処理を担当します。
    - `ContinueSymptomHandler.java`: 症状の複数入力（継続）を処理し、再びカテゴリ選択へ戻ります。
    - `NextToDateHandler.java`: 症状入力を終え、日付選択へ進む処理を担当します。
    - `SelectDateHandler.java`: 発生日付の選択を処理し、時間帯選択へ進みます。
    - `SelectTimezoneHandler.java`: 時間帯の選択を処理し、所要時間選択へ進みます。
    - `SelectDurationHandler.java`: 所要時間の選択を処理し、対象者選択へ進みます。
    - `SelectWhoHandler.java`: 主な対象者（父母など）の選択を処理します。
    - `ShowWhoSubHandler.java`: 対象者のサブメニュー（義父母など）を展開します。
    - `SelectWhoDetailHandler.java`: サブメニューからの詳細な対象者選択を処理します。
    - `SelectToWhoHandler.java`: 誰に対する行動か（自分、ケアマネなど）の選択を処理します。
    - `ShowToWhoSubHandler.java`: 行動対象者のサブメニューを展開します。
    - `SelectToWhoDetailHandler.java`: サブメニューからの詳細な行動対象者選択を処理します。
    - `WantMemoHandler.java`: 「メモ追加」を選択した際、テキスト入力待ち状態へ遷移します。
    - `SaveNoMemoHandler.java`: メモなしでの最終保存処理をキックします。
    - `SaveFinalHandler.java`: メモ入力後の最終保存処理をキックします。
    - `CancelRecordHandler.java`: 記録をキャンセルし、セッションを破棄します。
## Postbackの `data` フォーマット（設計書）

JSONの `data` 部は、Java側で `action=○○` というキーで分岐できるように、以下の命名規則で統一する。

| 現在のInputPhase | ユーザーの操作（Flex Message） | 送信する `data` の形式 | 次のInputPhase |
| :--- | :--- | :--- | :--- |
| **WAITING_FOR_SYMPTOM_CATEGORY** | 「困った」「記憶と行動」等を選択 | `action=select_symptom_category&categoryId=1` | **WAITING_FOR_SYMPTOM_DETAIL** |
| **WAITING_FOR_SYMPTOM_DETAIL** | 「同じことを何度も聞く」等を選択 | `action=select_symptom_detail&categoryId=1&symptomId=101` | **WAITING_FOR_LOOP_CHOICE** |
| **WAITING_FOR_SYMPTOM_DETAIL** | 「カテゴリ選択に戻る」を選択 | `action=back_to_category` | **WAITING_FOR_SYMPTOM_CATEGORY** |
| **WAITING_FOR_LOOP_CHOICE** | 「はい（続けて入力）」を選択 | `action=continue_symptom` | **WAITING_FOR_SYMPTOM_CATEGORY** |
| **WAITING_FOR_LOOP_CHOICE** | 「次のステップへ」を選択 | `action=next_to_date` | **WAITING_FOR_DATE** |
| **WAITING_FOR_DATE** | カレンダーで日時を選択 | `action=select_date` <br>*(※日付は `params` 等の別フィールドに入る想定)* | **WAITING_FOR_TIMEZONE** |
| **WAITING_FOR_TIMEZONE** | 「朝」「昼」等を選択 | `action=select_timezone&code=morning` | **WAITING_FOR_DURATION** |
| **WAITING_FOR_DURATION** | 「30分まで」等を選択 | `action=select_duration&id=1` | **WAITING_FOR_WHO** |
| **WAITING_FOR_WHO** | 「母」「父」等を選択 | `action=select_who&id=1` | **WAITING_FOR_TO_WHO** (※1) または<br>**WAITING_FOR_SAVE_OR_MEMO** |
| **WAITING_FOR_WHO** | 「配偶者/その他」を選択 | `action=show_who_sub` | **WAITING_FOR_WHO_SUB** |
| **WAITING_FOR_WHO_SUB** | 「義母」「祖父」等を選択 | `action=select_who_detail&id=11` | **WAITING_FOR_TO_WHO** (※1) または<br>**WAITING_FOR_SAVE_OR_MEMO** |
| **WAITING_FOR_TO_WHO** | 「自分（介護者）」等を選択 | `action=select_to_who&id=1` | **WAITING_FOR_SAVE_OR_MEMO** |
| **WAITING_FOR_TO_WHO** | サブメニューを開くボタンを選択 | `action=show_to_who_sub` | **WAITING_FOR_TO_WHO_SUB** |
| **WAITING_FOR_TO_WHO_SUB** | 「ケアマネ」「医師」等を選択 | `action=select_to_who_detail&id=11` | **WAITING_FOR_SAVE_OR_MEMO** |
| **WAITING_FOR_SAVE_OR_MEMO** | 「メモなしで保存」を選択 | `action=save_no_memo` | **（完了・DB保存）** |
| **WAITING_FOR_SAVE_OR_MEMO** | 「メモ追加保存」を選択 | `action=want_memo` | **WAITING_FOR_MEMO_TEXT** |
| **WAITING_FOR_SAVE_OR_MEMO** | 「キャンセルしてメニューに戻る」を選択 | `action=cancel_record` | **（セッション破棄・Startへ）** |
| **WAITING_FOR_MEMO_TEXT** | （キーボードで文字入力） | *(Postbackではなく通常テキストイベント)* | **WAITING_FOR_FINAL_CONFIRM** |
| **WAITING_FOR_FINAL_CONFIRM** | 「この内容で保存する」を選択 | `action=save_final` | **（完了・DB保存）** |
| **WAITING_FOR_FINAL_CONFIRM** | 「キャンセルしてメニューに戻る」を選択 | `action=cancel_record` | **（セッション破棄・Startへ）** |

*(※1) 選択した症状のカテゴリが「困った」の場合は `WAITING_FOR_TO_WHO` へ、それ以外の場合は `WAITING_FOR_SAVE_OR_MEMO` へ分岐する。*



  #### DTO（Data Transfer Object）
  * CareRecordDraft  /  PostbackRequestDto  /  CareRecordCreateCommand 
    * 各層間でデータをやり取りするためのオブジェクトです。リクエスト情報のカプセル化や、登録に必要なコマンドの転送に使用されます。
  ### 3. Domain層
  システムの核となるビジネスルールやエンティティを定義する層です。
  #### エンティティ・値オブジェクト
  * CareRecord  /  CareRecordDetail 
    * 介護記録の「ファクト（事実）」を表現するエンティティです。Append-Only（追記のみ）の原則に従い、データの不変性を担保します。
  * UserSession 
    * LINE上での対話状態（ステートマシン）や、一時的な入力データを保持・管理するためのエンティティです。
  * PeopleCare  /  SymptomName 
    * 介護の対象者や、記録する症状のカテゴリ・詳細を定義するマスタデータエンティティです。
  * LineWebhookEvent 
     * 処理済みのWebhookイベントIDを記録し、LINEプラットフォームからの重複リクエスト（冪等性）を制御するためのエンティティです。
  * JsonData ,  TimePeriod ,  TimeSpend ,  InputPhase 
    * データ構造や時間、入力フェーズを厳密に型付けするための値オブジェクトおよびEnumです。

  #### リポジトリ
  * CareRecordRepository  などの各種Repositoryインターフェース
    * ドメインオブジェクトの永続化および検索を行うためのインターフェースです
    * Spring Data JDBCを用いた実際の実装はInfrastructure層に委ねられます。
  ### 4. Infrastructure層 / Common
  技術的な詳細、設定、およびアプリケーション全体の共通処理を提供する層です。
  * JdbcConfig 
    * Spring Data JDBCのカスタム設定クラスです。PostgreSQLの JSONB 型やTIMESTAMPTZ 型とJavaオブジェクト間の変換（Converter）などを定義します。
  * SnowflakeIdGenerator / SnowflakeProperties 
    * 分散環境でも一意に特定可能なID（SnowflakeID）を生成するためのユーティリティおよび設定クラスです。
  * GlobalExceptionHandler  /  CustomAppException 
    * アプリケーション全体で発生する例外を捕捉・処理し、システムエラーや不正なリクエストに対する適切な応答を制御します。
  ### 5. ルートパッケージ
  - `CareRecordBotApplication.java`: Spring Bootアプリケーションの起動クラス。

## 状態管理と防衛線（セッション・フェーズ制御）

本システムは、ユーザーがLINE上で段階的に介護記録を入力していく「ステートマシン」として機能します。不正な操作や予期せぬ状態異常を防ぐため、以下の防御機構を備えています。

### 1. セッションの強制リセット機構（ユーザー体験の向上）
ユーザーが操作を途中で放置した場合など、セッションが「入力待ち」の状態で残ることがあります。
この状態でユーザーが新たに「開始」または「記録」と入力した際、システムは現在のフェーズ（状態）を問わず、**強制的に既存のセッション情報を初期状態（WAITING_FOR_SYMPTOM_CATEGORY）に上書き（UPDATE）**します。
これにより、ユーザーは「リセット」コマンドを打つ手間なく、いつでもスムーズに新しい記録を開始できます。

### 2. 「第二の盾」：フェーズチェック機構（不整合の防止）
`handlePostback` 処理内には、過去のメッセージのボタンが押された場合などに備え、「現在のセッションフェーズ」と「ハンドラーが許可するフェーズ」を照合する門番（第二の盾）が存在します。

* **通常時の防御**: 現在「対象者選択待ち」なのに過去の「症状選択」ボタンが押された場合、フェーズが一致しないためエラーとして弾き、データの不整合を防ぎます。
* **強制リセットとの完璧な連携**: ユーザーが「開始」と入力してセッションが初期化（フェーズが `WAITING_FOR_SYMPTOM_CATEGORY` に戻る）された直後に、古い「症状選択」ボタンが押された場合、現在のフェーズとボタンのアクションが「症状選択」で一致するため、正常な処理として受け入れられます。
このように、強制リセット機構はシステムの防衛線を破ることなく、合法的かつ安全に状態をリセットする手段として機能します。

## テストスクリプトの役割

### `./test-webhook.sh`
LINEプラットフォームからのWebhookリクエストをローカル環境で擬似的に再現するためのシェルスクリプトです。本物のLINEアプリやngrokなどのトンネリングツールを使わずに、バックエンドのControllerからService、DB保存までの結合テストをローカル単体で実行できます。

**【詳細な動作】**
- **引数の受け取り**: スクリプト実行時の第1引数（例: `./test-webhook.sh "開始"`）を、ユーザーがLINEで送信した「テキストメッセージ」として扱います。
- **JSONペイロードの組み立て**: LINEのMessaging API Webhookイベントと同じ構造のJSONデータ（`events[0].type = "message"`, `message.text = 引数の文字列` 等）を自動生成します。
- **署名（Signature）の生成**: `.env` ファイルに定義された `LINE_CHANNEL_SECRET` を読み込み、HmacSHA256アルゴリズムを用いてJSONボディの署名（Base64エンコード）を生成します。
- **HTTPリクエストの送信**: 生成したJSONと署名ヘッダー（`X-Line-Signature`）を付与し、ローカルのSpring Bootアプリケーション（`http://localhost:8080/api/v1/webhook/line`）に対して `curl` コマンドで `POST` リクエストを送信します。

**【用途例】**
- ステートマシンの状態遷移テスト（「開始」→「who=101」→「date=...」の連続送信）
- `LineWebhookDispatcherService` の対話ロジックの動作確認
- DB（PostgreSQL）への最終的なデータ永続化が正しく行われるかの結合テスト

#### 1.最初の方針（DDDの責領・違反チェック）との答え合わせ
- **設計意図**:
  - **ドメインロジックの非汚染**:
  DB（Infrastructure層）に新たなテーブルを追加することなく、`JwtTokenService`（Application層）のみで暗号化・復号化を完結させる究極のステートレス設計としました。
  - **責務の分離**: `RequestCsvHandler`はトークンを含めたURLを構築してLINEに返信する責務のみを負い、`CareRecordExportController`（Presentation層）は受け取ったトークンの検証と、認可されたユーザーへのデータ返却のみを行うよう役割を明確に切り離しています。
  - **【追記】Webhookの防衛機構 (`@LineMessageHandler`)**:
  本日の調査により、`LineWebhookController` には `line-bot-spring-boot-handler` SDK が提供する `@LineMessageHandler`が正しく配備されていることを確認いたしました。このアノテーションの裏側では
  、システムがリクエストを受け取る直前にフィルターが作動し、`X-Line-Signature` ヘッダーと `LINE_CHANNEL_SECRET`を用いた署名検証を自動で完遂しています。これにより、偽装されたリクエストがPresentation層に到達することを完全に防ぐ、極めて堅牢な作りとなっております。
  - **クリーンさの証明**:
  今回の修正においても、JPAの禁忌たるアノテーション（`@Entity`等）の混入や、システム規約で固く禁じられている `UPDATE` / `DELETE`メソッドの追加は一切行われておりません。設計思想を完璧に守ったまま、純粋なセキュリティ強化のみを達成しました。
    
### 2.エッジケースと未対応の課題
  - **想定した異常系**:
    - **有効期限切れのトークン**:
    JWTに「発行から10分間」という厳格な寿命（`exp`クレーム）を刻み込みました。10分経過後にアクセスされた場合は、ライブラリが自動で`JWTVerificationException` を送出し、`401 UNAUTHORIZED`として安全にリクエストを破棄（門前払い）します。
    - **トークンの改ざん**: 第三者が他人の `userId`に書き換えてアクセスしようとした場合でも、署名（HMAC256 + チャンネルシークレット）が一致しなくなるため、同様に検証エラーとして弾かれます。
    - **技術負債（Technical Debt）**:
      - **トークンの強制無効化の欠如**:
      JWTのステートレスな仕様上、一度発行したトークンをサーバー側から有効期限前に「強制的に破棄」する仕組み（ブラックリスト管理等）は用意していません（10分という短命な寿命によりリスクを最小化しているため、現状のフェーズでは許容範囲としています）。
      - **暗号鍵の流用**: 現在、JWTの署名用シークレットキーとして`LINE_CHANNEL_SECRET`を流用しています。テーブル追加や環境変数の追加というインフラ側の変更コストを避けるための合理的な判断ですが、将来的にシステムが巨大化した際は、JWT専用の独立した秘密鍵（環境変数）を用意し、鍵のローテーションが可能な体制を整えることが望ましいです。
