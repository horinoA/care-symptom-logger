# 詳細設計書 (フォルダ構成およびクラスの役割)

本プロジェクト（CareRecordDWH）は、DDD（ドメイン駆動設計）/オニオンアーキテクチャの思想に基づいて構築されています。依存関係は外側から内側（ドメイン）へ向かい、ドメイン層が他の層に依存しない設計となっています。

## フォルダ構成と役割

`tech.doshikawa.carerecord` 配下の主要なパッケージ構成は以下の通りです。

- **`presentation/`**: アプリケーションの入り口（外層）。LINEプラットフォームなどの外部からのリクエスト（Webhookなど）を受け取り、ビジネスロジックは持たずに `application/` 層に処理を委譲します。
- **`application/`**: ユースケースを実現する層。トランザクションの制御、ステートマシン（対話フロー）の進行、およびドメインオブジェクトを協調させてアプリケーションの目的を達成します。
- **`domain/`**: ビジネスロジックの核となる層（最内層）。データベースの構造や外部APIに依存せず、業務ルールそのものを表現する Entity、Enum（Type）、および Repositoryインターフェースを配置します。
- **`infrastructure/`**: 技術的な詳細を担当する層（外層）。Spring Data JDBCを用いたデータベースへのアクセス設定、外部API通信の設定など、泥臭い実装を担当します。
- **`common/`**: システム全体で共通して使用される設定クラス、ユーティリティ、例外処理クラスなどを配置します。

## 全クラスの役割一覧

### 1. Presentation 層 (`presentation.controller`)
- `LineWebhookController.java`: LINEプラットフォームからのWebhook（HTTP POST）を受け取るコントローラー。受信したイベントを `LineWebhookDispatcherService` に渡す役割のみを持ちます。

### 2. Application 層 (`application.*`)
- **Service**
  - `CareRecordApplicationService.java`: 介護記録の永続化処理を担当。マスタの存在チェックを行い、SnowflakeIDを採番した上で、親テーブルと子テーブルへ一括保存（Spring Data JDBC）を行います。
  - `LineWebhookDispatcherService.java`: LINEからのメッセージに応じたステートマシン（状態遷移）を管理。`UserSession` を使って現在のフェーズを判定し、適切な処理や返信メッセージの作成を行います。完了時に `CareRecordApplicationService` を呼び出します。
- **DTO**
  - `CareRecordCreateCommand.java`: 介護記録を生成するためのパラメータをまとめた不変オブジェクト（コマンド）。
  - `CareRecordDraft.java`: ユーザーがLINEで入力中の途中状態（ドラフト）を保持するクラス。このオブジェクトがJSONにシリアライズされ、セッションに保存されます。
  - `PostbackRequestDto.java`: LINEのPostbackアクション時に受け取るJSONデータをマッピングするためのDTO。

### 3. Domain 層 (`domain.*`)
- **Entity**
  - `CareRecord.java`: `event.care_records`（介護記録の親テーブル）に対応。主キーは手動発番のBIGINT。
  - `CareRecordDetail.java`: `event.care_record_details`（介護記録の子テーブル: 症状）に対応。`@MappedCollection` により親と紐付きます。
  - `PeopleCare.java`: `core.people_care`（対象者マスタ）に対応。
  - `SymptomName.java`: `core.symptom_name`（症状マスタ）に対応。
  - `UserSession.java`: `session.user_sessions`（セッションテーブル）に対応。Webhookの対話状態を永続化します。
- **Repository**
  - `CareRecordRepository.java`: `CareRecord` エンティティに対するCRUD操作を定義するインターフェース（Spring Data JDBC）。
  - `PeopleCareRepository.java`: `PeopleCare` マスタへのアクセスインターフェース。
  - `SymptomNameRepository.java`: `SymptomName` マスタへのアクセスインターフェース。
  - `UserSessionRepository.java`: `UserSession` エンティティへのアクセスインターフェース。
- **Type (Enum/ValueObject)**
  - `InputPhase.java`: LINEボットの現在の入力フェーズ（症状待ち、対象者待ちなど）を表す列挙型。
  - `JsonData.java`: PostgreSQLの `JSONB` カラムをJavaで扱うためのラッパークラス。
  - `TimePeriod.java`: 時間帯（朝、昼、夕、夜など）を表す列挙型。
  - `TimeSpend.java`: 経過時間（30分未満、2時間など）を表す列挙型。

### 4. Infrastructure 層 (`infrastructure.*`)
- **Config**
  - `JdbcConfig.java`: Spring Data JDBCのカスタム設定。PostgreSQL特有の型（`JSONB` 等）や `TIMESTAMPTZ`（`OffsetDateTime`）とJavaオブジェクトを相互変換するための `ReadingConverter` と `WritingConverter` を登録します。

### 5. Common 層 (`common.*`)
- **Config**
  - `SnowflakeProperties.java`: SnowflakeID生成に必要なワーカーIDやデータセンターIDなどを `application.properties` から読み込む設定プロパティクラス。
  - `ValidationConfig.java`: バリデーションに関する共通設定。
- **Exception**
  - `CustomAppException.java`: アプリケーション固有の例外の基底クラス。
  - `GlobalExceptionHandler.java`: Controller全体で発生した例外を捕捉し、統一されたエラーレスポンス（またはログ出力）を行うハンドラー。
  - `NotFoundMasterExceprion.java`: 存在しないマスタIDが指定された場合にスローされる例外。
- **Util**
  - `SnowflakeIdGenerator.java`: TwitterのSnowflakeアルゴリズムに基づいて、分散環境でも一意でソート可能な BIGINT (Long) ID を高速に発番するユーティリティクラス。

### 6. ルートパッケージ
- `CareRecordBotApplication.java`: Spring Bootアプリケーションの起動クラス。

## テストスクリプトの役割

### `./test-webhook.sh`
LINEプラットフォームからのWebhookリクエストをローカル環境で擬似的に再現するためのシェルスクリプトです。本物のLINEアプリやngrokなどのトンネリングツールを使わずに、バックエンドのControllerからService、DB保存までの結合テストをローカル単体で実行できます。

**【詳細な動作】**
- **引数の受け取り**: スクリプト実行時の第1引数（例: `./test-webhook.sh "開始"`）を、ユーザーがLINEで送信した「テキストメッセージ」として扱います。
- **JSONペイロードの組み立て**: LINEのMessaging API Webhookイベントと同じ構造のJSONデータ（`events[0].type = "message"`, `message.text = 引数の文字列` 等）を自動生成します。
- **署名（Signature）の生成**: `.env` ファイルに定義された `LINE_CHANNEL_SECRET` を読み込み、HmacSHA256アルゴリズムを用いてJSONボディの署名（Base64エンコード）を生成します。
- **HTTPリクエストの送信**: 生成したJSONと署名ヘッダー（`X-Line-Signature`）を付与し、ローカルのSpring Bootアプリケーション（`http://localhost:8080/api/v1/webhook/line`）に対して `curl` コマンドで `POST` リクエストを送信します。

**【用途例】**
- ステートマシンの状態遷移テスト（「開始」→「who=101」→「date=...」の連続送信）
- `LineWebhookDispatcherService` の対話ロジックの動作確認
- DB（PostgreSQL）への最終的なデータ永続化が正しく行われるかの結合テスト
